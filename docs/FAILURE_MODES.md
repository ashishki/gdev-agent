# Failure Modes And Error Taxonomy

This document is the canonical failure-mode taxonomy for the local portfolio hardening track. It
defines stable names that tests, logs, traces, and runbooks can reference. It does not claim a
production SLA or externally operated incident process.

## Taxonomy Contract

Stable names use the `FM_<DOMAIN>_<CONDITION>` pattern. New tests should assert these names in
documentation links, log/event expectations, or explicit error payloads where the implementation
exposes a code.

| Stable name | Trigger | Expected user-visible behavior | Logs, metrics, and traces | Retry and idempotency behavior | Operator response | Proof status |
|-------------|---------|--------------------------------|---------------------------|--------------------------------|-------------------|--------------|
| `FM_DUPLICATE_WEBHOOK_REPLAY` | Duplicate webhook replay: the same tenant sends a webhook with the same `message_id` within the Redis dedup TTL. | Return the cached webhook response with HTTP 200; do not create duplicate tickets, approvals, audit rows, or cost entries. | Trace span `middleware.dedup`; log events `dedup_hit` and `webhook_dedup_hit`; `gdev_webhook_service_calls_total{method="handle",outcome="dedup_hit"}`. | Idempotent replay. Redis key `{tenant_id}:dedup:{message_id}` stores the serialized response for 24 h. | Confirm repeated upstream delivery, inspect dedup hit rate, and avoid manual reprocessing unless the original response was wrong. | Covered by `tests/test_dedup.py` and workflow replay proof in `tests/test_webhook_service.py`. |
| `FM_REDIS_UNAVAILABLE` | Redis cannot be reached at startup, or approval/dedup Redis operations fail at runtime. | Startup fails fast if Redis is unavailable. Runtime approval/dedup failures surface as HTTP 5xx so n8n retries instead of losing correctness-critical state. | Startup raises `RuntimeError("Redis unavailable at startup")`; runtime logs include `webhook_dedup_lookup_failed` or approval-store exceptions; traces record the exception. | No idempotent success is claimed while Redis is unavailable. Upstream retries should wait until Redis recovers. | Restore Redis, verify key namespace health, rerun the failed webhook if upstream exhausted retries, and check pending approval queue consistency. | Covered by `tests/test_webhook_service.py` and `tests/test_redis_approval_store.py`. |
| `FM_REDIS_DEGRADED_RATE_LIMIT` | Redis operation inside `RateLimitMiddleware` fails after startup. | Request is allowed through instead of blocking all traffic; rate limiting is temporarily degraded. | Log event `rate_limit_bypass`; trace attributes `rate_limit.bypassed=true`; elevated request volume should be visible in request metrics. | Fail-open only for rate limiting. Dedup and approval storage remain correctness-critical and do not share this policy. | Restore Redis, watch abuse indicators, and consider temporary upstream throttling if traffic spikes while rate limiting is degraded. | Covered by `tests/test_middleware.py`. |
| `FM_POSTGRES_UNAVAILABLE` | Postgres connection or transaction fails for request, audit, budget, approval, or ticket persistence paths. | Return HTTP 5xx for paths that require persistence. Do not claim action execution if the ticket/audit write cannot complete. | Service logs include the failing service event, exception info, and trace exception; Postgres health should surface through infrastructure monitoring. | Upstream retries are allowed. Idempotency depends on whether the response was cached before the failure; correctness-critical writes should be verified before manual replay. | Restore Postgres, inspect failed request traces, compare pending decisions and audit rows, then replay only missing operations. | Covered by `tests/test_db.py` and `tests/test_redis_approval_store.py`. |
| `FM_POSTGRES_DEGRADED` | Postgres remains available but slow, saturated, or near connection exhaustion. | Requests may exceed local latency targets and can return HTTP 5xx if timeouts occur. No cross-tenant bypass or unsafe auto-execution is allowed. | `gdev_request_duration_seconds`, service duration histograms, DB connection count, slow query logs, and trace spans around persistence calls. | Retry only when callers receive a terminal 5xx/timeout. Do not duplicate manual approvals without checking `pending_decisions` and `approval_events`. | Check DB connections, locks, and slow queries; reduce background RCA load; scale Postgres only after confirming it is the bottleneck. | Covered by Postgres engine guardrail tests in `tests/test_db.py`; load proof planned later. |
| `FM_LLM_TIMEOUT` | LLM timeout: provider call times out or returns transient 5xx/overload errors after configured retries. | Return HTTP 5xx or safe escalation; do not auto-execute an action without a valid model result. | Trace span `llm.api_call` with `status=error`; `gdev_llm_requests_total{status="error"}` and `gdev_llm_retry_total`; provider exception in logs. | LLM client retries transient 5xx up to 3 attempts with exponential backoff. n8n can retry the webhook if the request fails. | Check provider status, retry saturation, budget impact, and whether deterministic demo mode can reproduce non-provider paths. | Covered by `tests/test_llm_client.py`. |
| `FM_LLM_MALFORMED_OUTPUT` | Malformed LLM output: tool output omits required fields, has invalid schema, or returns malformed category/entity values. | Fail closed to safe defaults and human review where risk is uncertain; never unsafe auto-approve because of malformed structure. | Log event `llm_invalid_response`; eval metric `invalid_structured_output_rate`; trace remains tied to `agent.llm_classify`. | No retry is required for deterministic schema failures unless provider behavior indicates a transient issue. The request continues only through safe fallback values. | Inspect prompt/model regression, compare eval gate output, and add cases before changing thresholds. | Covered by eval validators and `tests/test_guardrails_and_extraction.py`. |
| `FM_OUTPUT_GUARD_BLOCK` | Draft response contains a secret-like token, disallowed URL, or other output guard block condition. | Return HTTP 500 with internal guard failure and do not send the unsafe draft or execute the action. | `gdev_guard_blocks_total{guard_type="output",reason=...}`; span `agent.output_guard` with `blocked=true`; log context if redaction occurs. | Fail closed. Upstream may retry, but repeated blocks should stay blocked until prompt or configuration changes. | Treat as safety-critical. Inspect prompt, allowlist, model output, and eval unsafe-output cases before unblocking. | Covered by `tests/test_output_guard.py`. |
| `FM_APPROVAL_TTL_EXPIRED` | Approval TTL expiry: human approval arrives after Redis TTL or application `expires_at` boundary. | `/approve` returns HTTP 404 with `"pending_id not found"`; user/operator should see the action as expired rather than executed. | Log event `pending_expired`; approval-service metrics show error outcome; queue depth should decrease when expired entries are discovered. | Do not retry the expired token. Create a new pending decision from a fresh webhook if the action is still needed. | Tell the reviewer the token expired, inspect queue delays, and tune approval notification workflow or TTL only with documented trade-offs. | Covered by `tests/test_approval_flow.py` and `tests/test_redis_approval_store.py`. |
| `FM_CROSS_TENANT_APPROVAL` | Cross-tenant approval attempt: a JWT tenant or approval secret attempts to approve a pending action owned by another tenant. | Reject the approval and do not execute the action; the original tenant's pending action remains isolated. | Approval service spans carry `tenant_id_hash`; approval events should not be written for the wrong tenant; future tests should assert a stable rejection code. | No retry across tenants. Correct tenant credentials are required. | Investigate as a security event, check JWT claims and approval link routing, and preserve audit evidence. | Covered by `tests/test_approval_flow.py::test_tenant_a_cannot_approve_tenant_b_pending_action`, `tests/test_approval_flow.py`, `tests/test_approval_service.py`, and `tests/test_isolation.py`. |
| `FM_RATE_LIMIT_EXCEEDED` | Rate-limit exceedance: per-user webhook or auth attempt counters exceed configured RPM/burst thresholds. | Return HTTP 429. For webhooks, upstream should honor `Retry-After` or wait before one retry. | Rate-limit span has `rate_limit.blocked=true`; request metrics show HTTP 429; existing architecture boundary names `"Rate limit exceeded"`. | Retry only after the configured delay. Repeated 429s are expected protection, not application errors. | Check traffic source, tenant/user distribution, and whether limits are too strict for the demo or load scenario. | Covered by `tests/test_middleware.py`. |
| `FM_BUDGET_EXCEEDED` | Budget exceedance: tenant daily cost has reached or exceeded `daily_budget_usd` before an LLM call. | Return HTTP 429 with error code `budget_exhausted`; do not call the LLM or execute an action. | `gdev_budget_exceeded_total`; `gdev_budget_utilization_ratio`; span `agent.budget_check`; eval jobs can record `aborted_budget`. | Do not retry until the budget resets or an operator intentionally changes the budget. | Notify tenant admin, inspect cost ledger, decide whether to reset/increase the budget, and document the reason. | Covered by `tests/test_cost_ledger.py`, `tests/test_eval.py`, and `tests/test_eval_runner.py`. |

## Adversarial Tenant Boundary Scenarios

| Scenario | Expected behavior | Proof |
|----------|-------------------|-------|
| Tenant A attempts to read tenant B audit logs. | Audit read APIs use `request.state.tenant_id` from the signed JWT; tenant B rows are not returned to tenant A. | `tests/test_endpoints.py::test_tenant_a_cannot_read_tenant_b_audit_logs` |
| Tenant A attempts to approve tenant B's pending action. | Approval lookup is tenant-namespaced; no action executes and tenant B's pending decision remains intact. | `tests/test_approval_flow.py::test_tenant_a_cannot_approve_tenant_b_pending_action` |
| Webhook presents a missing, unknown, or invalid tenant slug. | Signature middleware rejects before downstream webhook handling. | `tests/test_middleware.py::test_missing_tenant_slug_rejected`, `tests/test_middleware.py::test_unknown_tenant_slug_rejected`, `tests/test_middleware.py::test_invalid_tenant_slug_rejected_predictably` |
| Webhook presents an invalid HMAC signature. | Signature middleware rejects before model calls, dedup writes, approval creation, audit writes, or other downstream side effects. | `tests/test_middleware.py::test_invalid_hmac_rejected_before_downstream_side_effects` |
